# ArmPi Ultra Robotics Project — Actionable Plan

> Generated by the `research-plan` workflow (12 agents: decompose → research → adversarial verify → synthesize), 2026-06-13.
> Web-researched and skeptic-checked, but still **verify each fact on the device** before relying on it.

A 6-DOF Hiwonder ArmPi Ultra on a **16GB Raspberry Pi 5 (no GPU)**, running ROS 2 Humble inside Hiwonder's Docker image. (Note: an earlier draft of this plan assumed a 4GB Pi — corrected to 16GB, so RAM is no longer a binding constraint; the **no-GPU / CPU-bound** limit still is.) Goal: voice → LLM → vision → IK → grasp. Steps are ordered so that each unblocks the next; the early "verify on device" steps gate everything downstream.

---

## Phase 0 — Bring-up & baseline (do first; everything branches on these facts)

1. **Power on and connect, all from inside Hiwonder's container.** 7.5V/6A to the STM32 board, wait ~60s for LED1 blink + beep; join `HW…` AP (pwd `hiwonder`), VNC/SSH to `192.168.149.1` as `pi/raspberrypi`.
   - **Caveat:** the host Pi OS has **no ros2**. Run every `ros2` command in Hiwonder's "ROS2 Environment Terminal" (auto-enters the container). Reproducibility comes from pinning **Hiwonder's customized SD image version/hash**, NOT stock `ros:humble`.

2. **Capture the regression baseline.** `~/.stop_ros.sh` → `ros2 launch sdk armpi_ultra.launch.py` → in a second container terminal record `ros2 node list`, full `ros2 topic list`, and `ros2 node info /ros_robot_controller`. This captured graph is your regression reference for all later work.

3. **Find the REAL camera topics — do not trust `/depth_cam/*`.** The camera is a **Deptrum Aurora930 Pro** on the **AScamera (Angstrong)** driver. Discover with `ros2 topic list | grep -i ascamera`; RGB is `/ascamera/camera_publisher/rgb0/image`, depth under the matching `depth0/…`. Record **separate** `ros2 topic hz` for RGB and depth (`hz` on `/depth_cam/rgb/image_raw` hangs forever).
   - **Caveat:** depth is **sensor-capped at 640×400 @ ~12fps** regardless of CPU.

4. **Liveness-check the 6 servos safely, then derive the real servo map.** 6 servos, IDs 1–6, range 0–1000, **ID1 = gripper**. The per-ID model map is NOT documented — derive it on-device from the SDK URDF/servo config. Use **small relative moves from home**, not raw `position: 300` into every joint (mixed servo models → different angle scales/limits).

5. **Resolve the serial path inside the container.** On Pi 5 the STM32 is on the **GPIO-header UART → expect `/dev/ttyAMA*`, NOT `/dev/ttyUSB0`**. Confirm via the `ros_robot_controller` launch parameter and verify device passthrough into the container.

## Phase 1 — Vision & perception

6. **Reuse factory camera calibration; verify alignment before re-calibrating.** Confirm the Aurora930 Pro driver publishes **hardware-aligned depth-to-color + a registered CameraInfo** (may need vendor build flag `STREAM_SDK_TYPE=AURORA930`). Overwriting factory intrinsics can break vendor depth↔color registration.

7. **Detector: pick framework by target FPS — NCNN's real speed is ~3–4 FPS @640, not 12.** (The 12 FPS/80ms figure is the **OpenVINO** row.)
   - ~12 FPS → export **YOLO11n to OpenVINO**, or MNN (~8.6 FPS).
   - Keep NCNN → drop `imgsz` to **320** for ~8–12 FPS.
   - `-seg` runs at ~half detect rate (2–5 FPS) — budget separately.
   - Only buy a **Hailo-8/8L HAT** (~$70–110, not in kit) for moving-object tracking.

8. **Deproject with a depth patch, not a single pixel.** Detection center/centroid → read aligned depth over a **small median patch**, reject zero/invalid returns, deproject. The 8mm accuracy spec is **at 1m** (3mm@0.5m) and degrades with distance/ambient IR — validate at YOUR working distance.

9. **Hand-eye calibration: quality over count.** Eye-on-base, **12–15 samples with maximal rotation diversity (~90° about each axis, both directions)**; "5+" is too few for an 8mm task. `easy_handeye2` is a community port (v0.5.0) — pin the commit, or use MoveIt 2's built-in `hand_eye_calibration` plugin (don't mix the two). Publish into `base_link` via tf2.

## Phase 2 — IK, motion planning, grasp

10. **Launch vendor MoveIt config and resolve three unknowns FIRST.** `ros2 launch armpi_ultra_moveit_config demo.launch.py`; inspect `kinematics.yaml`/SRDF/`joint_limits.yaml`. Steps 11–13 branch on: (a) ros2_control gripper controller vs raw servo topic, (b) which IK plugin ships, (c) real Pi-5 IK/plan latency.

11. **Fix the actual IK blocker: 5-DOF, not the solver.** The arm is a **5-DOF chain + gripper** — a full 6-D `PoseStamped` goal is over-constrained and will `IK_FAIL` on most targets *regardless of solver*. Set **`position_only_ik: true`** and design grasps around a fixed top-down / fixed-yaw approach.
    - **TRAC-IK caveat:** there is **no `ros-humble-trac-ik-kinematics-plugin` apt binary**. Either build from source (`aprotyas/trac_ik @ ros2`, pulls NLopt) or **stay on KDL + `position_only_ik`** (often adequate). If using TRAC-IK, set `solve_type: Speed` with a 0.005–0.02s timeout.

12. **Build the pose→result action server with pymoveit2 — clone + colcon, not pip.** Clone `AndrejOrsula/pymoveit2` (Humble branch) into your workspace overlay. Node takes `PoseStamped`, plans+executes, returns success + reason (`IK_FAIL`/`PLAN_FAIL`/`EXEC_FAIL`).

13. **Gripper: prefer GripperActionController, plan for raw-topic fallback.** If no ros2_control gripper controller is exposed, publish servo **ID1** on `/ros_robot_controller/bus_servo/set_position` (close=low, open=high), gating on `/ros_robot_controller/init_finish`. The raw topic is **outside MoveIt's collision model** and has **no force feedback** — add stall/effort safety so a 25kg serial servo doesn't overheat on a grasped object.

14. **Planning config:** RRTConnect, 1–2s planning budget, single-IK reachability pre-check. Keep **FCL** (Humble default) collision checking — do NOT swap to Bullet as a "speedup" (not thread-safe, false positives on non-convex meshes).

## Phase 3 — Voice front-end & LLM

15. **Wake word: openWakeWord with onnxruntime, NOT tflite.** `pip install openwakeword onnxruntime` into a virtualenv, set `inference_framework='onnx'`. The default tflite-runtime wheels are Python-3.11-only and **fail to install on Humble's Python 3.10**. Treat openWakeWord as maintenance-frozen (last release 0.6.0) — vendor/pin it, document a Porcupine/Vosk fallback.

16. **LLM: cloud is the quality/latency default, but local is now viable on 16GB.** With 16GB RAM you can *fit* a quantized 3B–7B local model (e.g. Qwen2.5-3B/7B-instruct via Ollama/llama.cpp) — the old "impractical" verdict was RAM-driven and no longer applies. The remaining limit is **CPU throughput**: no-GPU Pi 5 inference is ~1–5 tok/s, so a short JSON action still takes several seconds.
    - **Use cloud (OpenAI/Claude API)** when you want best reasoning + lowest latency; plan for recurring **API cost**, **network/rate-limit failures** (need a recovery branch, not just "ambiguous→clarify"), and **privacy** (voice/intent leaves the device).
    - **Use local** when you want zero cost / offline / private and can tolerate slower, weaker reasoning. Your current `qwen:0.5b` is too weak for reliable JSON intent — step up to a **3B-class quantized** model now that RAM allows it.

17. **Keep voice off the real-time control path; set honest latency expectations.** End-to-end **8–25s** on a no-GPU Pi 5 (STT + reasoning + TTS), dominated by CPU compute, not RAM. With 16GB the STT+LLM+TTS stack no longer risks OOM against the ROS 2 control nodes, so you can run the full pipeline on-device — just don't put voice on any latency-sensitive control loop.

## Phase 4 — Orchestration, integration, testing

18. **Pick the behavior-tree library ONCE: `py_trees_ros` (Humble 2.2.x).** Lower-friction since the LLM glue and SDK are Python. "Port to BehaviorTree.CPP later" is a **rewrite in another language**, not a config flip. Don't hand-roll a raw state machine.

19. **Wrap each capability as a ROS 2 action server** (`vision_detect`, `arm_pick`, `arm_place`, `arm_home`) for feedback/cancel/timeout. **Tree:** `WaitForVoiceCommand → ParseIntentWithLLM → DetectObject → PlanAndPick → Place → Report`, with `Fallback`/`RetryUntilSuccessful`/`Timeout` decorators and recovery subtrees.

20. **Fix the timeout knob.** The action-server `result_timeout` default in Humble is **15 minutes** (don't touch it). The real Pi-5 gotcha is **`BT::RosActionNode`'s `server_timeout`/`wait_for_server_timeout`** — raise these above your slowest measured CPU-only MoveIt plan (often 20–60s).

21. **Split sim/CI from runtime.** Run **launch_testing + Gazebo + Groot2 on a dev PC/CI**; run only the trimmed `driver + vision + moveit + voice + coordinator` graph on the Pi. (On 16GB this is now a **CPU/thermal** choice, not an OOM one — Gazebo + MoveIt + YOLO together will still saturate the 4 cores and thermally throttle, so keep heavy sim off the Pi.) CI: `colcon test --packages-select <pkg>` + `colcon test-result --all --verbose`.

## Key trade-offs at a glance

- **Local vs cloud:** Vision + IK + wake-word stay **local** (CPU-bound). LLM can go **either way** on a 16GB Pi: cloud (OpenAI/Claude) for best reasoning + lowest latency, or a local 3B-class quantized model for zero-cost/offline/private (slower, weaker). Don't stay on `qwen:0.5b`.
- **NCNN vs OpenVINO:** OpenVINO ~12 FPS vs NCNN ~3–4 FPS @640. Use OpenVINO (or NCNN@320).
- **16GB Pi 5:** RAM is not a constraint — the full STT+LLM+TTS+ROS stack fits. The binding limit is **CPU/thermal** (4 cores, no GPU): still never run RViz/Gazebo on the Pi in production.
- **KDL vs TRAC-IK:** the 5-DOF `position_only_ik` fix matters far more than the solver.
- **FCL vs Bullet:** keep FCL default.
- **Hailo HAT:** skip for static pick-and-place.
